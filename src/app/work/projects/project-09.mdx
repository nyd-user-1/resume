> **Version 1.2** — published 2026-06-18. HQ v1.1 framed itself around a limit: it could *watch* a Claude Code session but never *drive* one. Three days later that limit was gone. This is the revised case study; the original observe-only writeup is preserved as [HQ v1.1: The Cockpit That Only Watched](/blog/hq-observe-only-cockpit).

## Overview

If an AI agent is doing the building, what does the cockpit look like? HQ is one answer. It is a localhost-only control layer that sits on top of **Claude Code** — the agentic coding tool used to build every other project in this portfolio — and turns the trail it leaves on disk into a working dashboard: which sessions are live, what each one costs, what shipped today, and what the assistant is doing right now.

It started from one observation. Claude Code already writes a complete, structured record of its own work to disk — every transcript, every token count, every session lineage — but there was no single place to *see* it. HQ became that place. Then it went one step further: HQ no longer just reads that record, it **drives the session that writes it.** A warm Claude process, owned by HQ, lets you steer a live conversation — or spin up a brand-new one — without ever opening the terminal.

The design rule stays deliberately strict. HQ has **no database, no authentication, no deployment, and no external services** — three runtime dependencies in total (`next`, `react`, `react-dom`). Everything else is `node:fs` over the files Claude Code already maintains: the JSONL transcripts under `~/.claude/projects`, the memory notes beside them, and `git log` across the `~/code` workspace. The disk is the database. Claude Code is the writer; HQ is the reader — and, when you ask it to, the writer's hands.

## Key Features

- **Drive from HQ — the live REPL**: The headline of v1.2. HQ now owns a warm `claude` process per session and speaks its bidirectional `stream-json` protocol directly, so the center column stops being a mirror and becomes a cockpit. You type into a Claude-style send box; turns stream back token by token; tool-permission prompts escalate out of the process and into the browser as **Approve / Deny** cards, with the verdict riding back to the waiting turn. One click births a brand-new session and drives it from the same seat — no terminal, no context loss. The safety contract is explicit: **one active writer at a time.** HQ drives while the TUI yields; stop the REPL and it hands the session back.
- **The always-on Terminal**: That cockpit is mounted exactly once and never remounts as the rest of the interface navigates around it — the persistent heart of the layout, the protected column other panels yield space to. The header reads `● project · session · cache · ctx`, where `ctx` counts down the percentage of the 1M-token context window still left, mirroring the CLI; the border colors live with turn state — thinking, done, interrupted.
- **Recent Sessions sidebar**: Every transcript touched in the last seven days, grouped by project or date, each with favorite / hide / rename behind a kebab menu (stored in a sidecar, never written back into Claude Code's own files). Session lineage is reconstructed across `/clear` boundaries so a conversation that was cleared and resumed reads as one chain.
- **Usage, Burn & Savings**: A token meter that turns raw transcript usage into a readable ledger — this session, the last 24 hours, the last 7 days, and a forward **burn forecast** — with the long-context premium past the 200k-token price cliff folded in. Alongside it, **Efficiency Mode** keeps a running tally of dollars saved and flags context bloat. It stays **measurement-only by design**: it reports and advises, it never reaches into a live turn to act.
- **Batch Planner**: New in v1.2 and the natural sequel to driving sessions. It reads the To-Do store, groups tasks into execution batches, and forecasts each batch's cost (a p50/p90 range calibrated from real cold-context token counts on disk) and its throughput against the current reset window — a live "*X tasks before reset*" figure. Drag a batch into the terminal to run it. It plans the work; you still pull the trigger.
- **Unified Search**: One surface over every corpus on disk — thirteen in all, anchored by a persisted ~17MB transcript index and reaching down to memory notes, saved blocks, components, and commits — with scope chips, an in-panel reader, and a relevance/recency sort. It makes months of AI sessions searchable the way a codebase is.
- **Components**: A self-documenting design-system registry. Each card renders its own provenance (`● claude · file · kind`), its live source as the body, and a footer tracing which session created it — drag a card into a terminal to drop its file path, or reorder cards to persist a display order.
- **Shipped**: A cross-project commit feed assembled from `git log` across every `~/code` repository; click any commit to read its diff in-panel, with chat-mentioned SHAs auto-linked to the same view.
- **And the rest of the cockpit**: an HQ-native To-Do store with per-session attribution; a Skills launcher that discovers every installed `/skill`; a Memory Audit over what loads into context each turn; a Firehose that dumps a transcript in full fidelity; and a click-to-copy CMD reference for the CLI's utility commands.
- **The Boundary**: A small but defining touch — every region is wrapped in a dashed box whose top border carries a click-to-copy chip showing the exact source file that renders it. The interface narrates its own anatomy, so the dashboard doubles as a map of its own code.

## Technologies Used

- **The live REPL (`lib/repl.ts`)**: HQ's write half. It spawns and keeps alive one `claude` subprocess per driven session — warm across turns — and speaks the bidirectional `stream-json` protocol: user turns as JSON lines on stdin, streaming events on stdout. Tool permissions escalate through a small local MCP shim that long-polls an HQ route until the browser answers. This is the machinery that turned HQ from an observer into a controller.
- **Next.js (App Router, Turbopack)**: Server components read the `lib/*` file engine directly and stream the result; client shells hold only interaction state. The right-hand panel is a single **parallel route** (`@panel`) so panels open and close without ever disturbing the persistent terminal beside them.
- **`node:fs` as the entire data layer**: No ORM, no client, no migrations. `lib/transcript.ts` parses turns, tool timelines, live status, and lineage; `lib/usage.ts` and `lib/pricing.ts` turn token counts into dollars; `lib/sessions.ts`, `lib/search.ts`, `lib/shipped.ts`, `lib/components.ts`, and `lib/planner.ts` each read one slice of disk. HQ-native state (favorites, to-dos, component order, notes) lives in small JSON/Markdown sidecars under `~/.claude/hq/`.
- **Server-Sent Events**: The REPL streams a live turn back into the UI event by event, and a short poll reconciles the session list against the truth on disk.
- **React 19 + Tailwind v4**: A three-column shell — sidebar, persistent terminal, panel portal — where the terminal is mounted once and protected. The component library follows a strict naming taxonomy (`[Category][Descriptor][Element]`, kebab-case files, PascalCase components) and is registered and rendered back to the user as the Components panel.

## Challenges and Learnings

The defining challenge inverted between versions. v1.1's hardest constraint was a ceiling: a headless `claude -p` call is one-shot — it can dispatch a turn but cannot *steer* a session, so HQ stayed an instrument that observed precisely and dispatched cleanly. The honest move at the time was to accept the limit rather than fake control.

v1.2 is the story of removing it. Driving a real session from outside the CLI meant owning a warm `claude` process and speaking its `stream-json` protocol in both directions — and, harder, handling the moment a turn asks permission to run a tool. A subprocess has no UI; the human is in the browser. The answer was to escalate: a local MCP shim intercepts the permission request, long-polls an HQ endpoint, and blocks the turn until the browser renders an Approve / Deny card and sends a verdict back. The turn then resumes as if the CLI had asked. The second hazard was correctness, not plumbing — two writers on one session corrupt it — so HQ enforces a single rule: **one active writer at a time.** When HQ drives, the TUI yields; stopping the REPL releases the session back. The limit named in v1.1 didn't fall to brute force; it dissolved once the boundary was drawn at the right place.

Some lines were held deliberately. **Efficiency Mode still only measures** — it will tell you a compaction would save a few dollars a turn, but it won't reach in and do it; the **Batch Planner forecasts** cost and throughput but never auto-executes; and because cost is a headline number, pricing is labeled honestly as an estimate, with the post-200k long-context premium modeled explicitly rather than hidden. Driving a session is a deliberate act the human initiates, not something HQ does behind your back.

The through-line survived intact: the most useful tooling for an agentic workflow didn't require new infrastructure — it required *reading what the agent already records*, presenting it well, and, only where it was safe, writing back through the same door. A ~17MB on-disk index made months of conversation searchable; a `git log` sweep made every repository's progress one feed; a dashed box with a file-path chip turned the UI into its own documentation; and a warm subprocess turned the dashboard into a cockpit.

## Outcome

HQ is a working answer to the question the rest of this portfolio kept raising: if an AI agent is doing the building, what does the cockpit look like? v1.1 proved a useful control layer could be built from almost nothing — three runtime dependencies and the files the agent already writes — and still surface real signal: live session status, true cost in dollars, a searchable memory of every conversation, a self-documenting component library, a cross-project record of what shipped. v1.2 earns the rest of the title. The same three-dependency, no-server tool now **drives** the agent it watches — births sessions, steers live turns, routes tool permissions through the browser, and forecasts the cost of work before you spend it — without giving up the rule that made v1.1 trustworthy: it acts only when you tell it to, and only as one writer at a time.

That is the small thesis underneath it. The agent's own footprint on disk is a rich, queryable substrate; the job of good tooling is to read it honestly, show it clearly — and, once the boundary is drawn carefully enough, to write back through the very same door. The cockpit no longer just watches the agent fly. It takes the stick.
