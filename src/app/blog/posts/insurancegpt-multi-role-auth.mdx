## Why I Didn't Use a Framework

NYSgpt had auth, but a simple one — essentially one user type with a single session model. InsuranceGPT, my third project, was the first where different user types needed completely different UI surfaces from the same codebase. That introduced a three-role model that made framework auth more painful than useful.

The three roles — consumer, agent, admin — have different session durations, different data access levels, and different login entry points. Consumers log in from the public chat interface and get a 7-day session. Agents log in from a separate `/agent` route and get a 24-hour session. Admins log in from `/admin` and get a 4-hour session with no refresh, backed by a bcrypt hash rather than a stored password.

Mapping that to Clerk would require configuring three separate organizations or user metadata fields, then writing role-checking middleware Clerk doesn't ship. The configuration overhead approaches the complexity of a custom implementation, plus MAU-based billing and vendor lock-in. The custom implementation is under 400 lines across three files: `auth.js` (bcrypt compare + JWT issue), `session.js` (cookie read/write + `safeGetUser`), and the three auth routes. Every subsequent project that needed multi-role auth referenced this one.

## The RequireAuth Guard

Each role has its own `RequireAuth` wrapper component. `<RequireConsumerAuth>` checks for a valid consumer JWT in the session cookie and redirects if absent. `<RequireAgentAuth>` does the same but also checks the `role` claim in the JWT payload — an agent JWT won't pass a consumer check, and vice versa. `<RequireAdminAuth>` additionally validates that the token was issued within the last 4 hours regardless of its own expiry field.

The guards are React components rather than Next.js middleware because the role-checking logic references the JWT payload, not just the cookie's presence. On consumer-facing pages I added a skeleton loader during the session check so the transition doesn't look broken.

## Role-Based Sidebar From One Component

The sidebar renders from a single `Sidebar` component that takes the current user's role as a prop and filters a static config object:

```js
{ label: 'Dashboard', href: '/admin/dashboard', visibleTo: ['admin'] },
{ label: 'Lead Console', href: '/agent/leads', visibleTo: ['admin', 'agent'] },
{ label: 'My Quotes', href: '/quotes', visibleTo: ['consumer'] },
```

No per-role sidebar component, no conditional rendering scattered through the tree. Adding a new navigation item means adding one entry to the config with a `visibleTo` field.

## Password Reset Flow

The password reset flow uses a token stored in the database rather than encoded in the JWT. The server generates a cryptographically random 32-byte token, stores its SHA-256 hash in `password_reset_tokens` with a `(user_id, expires_at)` record, and sends the raw token in an email link. The window is one hour. On use, the row is deleted — single-use prevents replay.

Agent welcome emails use the same token mechanism for initial password setup via a one-time link.

## The safeGetUser Guard

`safeGetUser()` wraps the JWT decode + localStorage read in a try/catch and returns `null` on any error. It exists because localStorage is writable by any script, and a corrupted session object will throw rather than return nothing. Without the guard, a parse error propagates up and crashes the render tree. With it, the `RequireAuth` wrapper sees `null`, treats the user as unauthenticated, and redirects to login. The pattern generalizes: read structured data from external sources defensively and treat parse errors as absence rather than exceptions.
