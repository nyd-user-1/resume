## Why Lineage Matters

When I look at the component tree of any app in the GPT suite, I can trace most shared components back to a specific moment in a specific project where they became generic enough to copy. That lineage isn't historical trivia — it tells me which components have been stress-tested across multiple contexts and which ones are still domain-specific and liable to break if someone tries to extract them.

A component that has been copied twice and survived both times has passed a meaningful generalization test. It means the API is flexible enough for different data shapes, the visual design is abstract enough to not look wrong in a different brand context, and the performance characteristics are acceptable across different page loads. That's more assurance than any amount of upfront design work provides.

The sequence of the six apps matters for understanding where each component came from: NYSgpt (the mothership, Jan–Feb 2026) → NSRgpt (Feb–Mar 2026) → InsuranceGPT (Mar 2026) → SolarGPT (Mar–Apr 2026) → ChildcareGPT (Apr 2026) → SportsGPT (Apr–May 2026). Components flowed in that direction, each fork inheriting what had survived and contributing new patterns that subsequent forks could copy.

## The Index+Detail Pattern: Born in NYSgpt

NYSgpt is where the core index+detail pattern was established. Bills, members, and committees pages gave the suite its first real component vocabulary: a filterable index table, a detail page with labeled metadata sections, breadcrumb navigation, and a sidebar. These weren't design system components — they were production pages that happened to establish conventions.

NSRgpt, the first fork, copied the index+detail structure directly. It added two things of its own: a more sophisticated citation system in the AI chat panel, and a live stream panel to the right of the main container. Semantic search layered on top of keyword search was NSRgpt's other contribution. Each of these became patterns subsequent apps could reference.

## InfoTable: Crystallized in InsuranceGPT

InfoTable existed in embryonic form in NYSgpt — bill metadata needed labeled rows grouped by section. But the component's final, portable design crystallized in InsuranceGPT. The InsuranceGPT detail page established the template: an ~866px container, H1 with a button group opposite (three buttons with left/right chevrons), breadcrumbs, a hero image at 866×425px, then H2 headings each followed by an InfoTable, repeated for two to three sections, with Card Carousels at the end. That structure — hero, then labeled fact tables, then browsable card rows — became the detail page blueprint for every app that followed.

The InfoTable in that context needed to handle insurance-specific data: coverage terms, policy limits, provider details, pricing tiers. Labeled rows, grouped sections, null-safe display, responsive accordion collapse. When SolarGPT needed to show electricity profiles — generation capacity, emissions rates, grid participation by fuel type — the InfoTable from InsuranceGPT translated directly after swapping the domain-specific types for a generic `{ label: string; value: string | number | null; group?: string }` row structure.

ChildcareGPT adopted the same component for facility profile accordion cards. A childcare facility profile has the same structure as an insurance product profile: labeled fields grouped into sections (licensing, capacity, contact information, inspection history), null-safe display, responsive layout. NYSgpt used InfoTable for bill metadata panels (sponsor, chamber, status, session, last amended). NSRgpt used it for filing metadata in document panels. The component appears in five of the six apps in a form that is structurally identical to the InsuranceGPT version — only the data passed in changes.

The lesson from InfoTable's spread is that labeled-row-in-grouped-sections is a pattern that appears in almost every domain-specific data application. Government data especially tends toward this structure. Build InfoTable once in the right shape, and you have the display layer for most structured domain records.

## RevealFx: Born in SportsGPT, Immediately Portable

RevealFx is an entrance animation: elements reveal themselves into the viewport by animating a clip-path from bottom to top while simultaneously fading from blurred to sharp. It's subtle enough not to feel gratuitous but present enough to give detail pages a sense of materialization rather than appearing all at once.

The original version in SportsGPT was hardcoded to the team hero section. An Intersection Observer watched the hero container, and when it entered the viewport, it toggled a CSS class that triggered the animation via a `transition` on `clip-path` and `filter: blur()`. That was it — one component, one use case, no props.

The first copy happened for SolarGPT county detail pages. County pages have a similar hero-area-plus-stat-sections structure, and the reveal animation felt right there too. But county pages have multiple independent sections that should stagger — the emissions chart shouldn't reveal at the same time as the generation chart. That forced the first generalization: a `delay` prop, passed as a CSS custom property so the transition-delay could be set per-instance without JavaScript.

The second copy, for ChildcareGPT facility pages, revealed another gap: some elements should only animate once (the hero), while others should re-animate if the section's content changes (the stat tables, which update when the user changes the selected season). That led to a `replayOnChange` prop that detaches and re-attaches the Intersection Observer when its dependency value changes.

By the time RevealFx reached NSRgpt and NYSgpt's detail pages, the API was stable: `children`, `delay`, `className`, `replayOnChange`. No subsequent app has needed anything beyond those four props. That stability is how I know the abstraction is correct — it found the right boundary on the second copy and hasn't shifted since.

## HeroSlider: Team Pages to Everywhere

HeroSlider is an auto-advancing photo slider with manual dot navigation. In SportsGPT, it was built for team pages to cycle through action photos of a program. The original had hardcoded timing (4 seconds per slide), a fade-only transition, and no touch support.

Division pages in SportsGPT needed HeroSlider too, but with different sizing — division pages have wider hero areas that don't fit the team page's aspect ratio. That forced a `aspectRatio` prop and made the slide container height-flexible rather than fixed.

SolarGPT's state profile pages adopted HeroSlider for a different purpose: cycling through region overview images (power plants, transmission infrastructure, urban and rural grid photos). The content category changed completely, but the component worked without modification beyond passing different images. That's the best kind of test for a component — using it for a purpose its author didn't intend, and discovering it handles it correctly.

ChildcareGPT uses HeroSlider for county overview pages, which show community photos rather than sports action shots. NYSgpt uses it on district profile pages. In each case, the images changed and the component didn't.

## CardCarouselPortrait and the Home Page Pattern

CardCarouselPortrait is a horizontally scrollable card row for entity browsing — teams on SportsGPT's home page, utility companies on SolarGPT's landing section, county overviews on ChildcareGPT. It emerged from SportsGPT, where team cards needed to be browsable without a full table or grid.

The component is generic at its core: it receives an array of items and a render function for each card. The card shapes are different in every app, but the carousel container, the scroll behavior, the fade-out on edges, and the responsive snap points are identical. When NYSgpt needed a committee browsing carousel, I passed in committee cards and it worked on the first try.

## AppLogo and app-config.ts: The Configuration Layer

The most important component in the shared suite isn't a visual component — it's the configuration object. Every app has an `app-config.ts` that defines its name, tagline, accent color, nav items, footer links, and logo component. The sidebar, the top nav, the footer, and the metadata all consume `app-config.ts` rather than having domain values hardcoded.

This means swapping an app's identity is literally a config file change. The shared layout components don't know what app they're running in — they know what app-config says. When I forked NYSgpt to build NSRgpt, the first hour was updating `app-config.ts`. The shared components inherited the new identity automatically. The same pattern repeated for every subsequent fork.

AppLogo follows the same pattern: it's a component that renders a different SVG or image based on the config, but the import point in the layout is always `AppLogo`. No layout file contains a hardcoded app name or logo path.

The lesson here is broader than configuration: components that survive copying are components that externalize their domain assumptions rather than embedding them. RevealFx externalizes timing. InfoTable externalizes data shape. AppLogo externalizes brand. The more assumptions a component embeds, the fewer contexts it can survive in.
