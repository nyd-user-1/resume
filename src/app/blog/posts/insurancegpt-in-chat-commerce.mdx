## The Navigation Problem in E-Commerce

Most e-commerce flows have the same shape: browse on the product page, redirect to a cart, redirect to a checkout page, redirect to a confirmation page. Each redirect is a context break — the user loses the reasoning that got them to "buy." They've left the decision surface and entered a transaction surface, and the two surfaces share almost no information. By the time they reach the confirmation page, they've forgotten why they chose this carrier over the others.

The standard justification for this pattern is separation of concerns. The product page is about discovery; the checkout page is about payment. Keep them separate.

That justification holds for physical goods with large catalogs and complex fulfillment logic. For insurance — where the "product" is a coverage configuration, the "seller" is an AI recommendation, and the "purchase" is a digital transaction that completes in seconds — it doesn't hold. The user needs the recommendation visible while they're deciding to buy. Redirecting them away from it is friction with no compensating benefit.

## The CheckoutDrawer

The solution is the `CheckoutDrawer`: a slide-up panel that renders the Stripe payment form inline, anchored to the bottom of the chat interface. It opens when the user clicks "Purchase" on a carrier recommendation card. The recommendation card stays visible behind the drawer — carrier name, premium amount, coverage details — so the user can reference it while filling in their payment information.

The drawer renders a Stripe `PaymentElement` mounted to a `PaymentIntent` created server-side the moment the user clicks "Purchase." The server call to create the PaymentIntent is fast enough (under 300ms typically) that I don't need an intermediate loading state — the drawer opens and the Stripe element is ready by the time it finishes animating in.

On payment completion, the drawer shows a confirmation state inline before closing. The chat interface then receives a `PURCHASE_CONFIRMED` event and appends a confirmation message to the conversation thread — making the purchase part of the conversation history rather than an external event that happened somewhere else. The user's session now contains the full arc: their vehicle, their coverage preferences, the AI's recommendation, and the purchase confirmation.

## Three AI Context Modes

The chat interface has three rendering modes that correspond to three user roles: consumer, agent, and quote.

In consumer mode — the default for unauthenticated users — the AI responds to natural language questions about vehicle safety, coverage types, and carrier comparisons. It doesn't show pricing or a purchase button. The goal is education and lead capture.

In quote mode — triggered when the user has provided vehicle, driver, and coverage details — the AI switches to a structured comparison view: a table of carriers with premiums, deductibles, and coverage highlights, with a "Purchase" button on the recommended option. The underlying AI call is the same; the response is rendered differently based on the context flag.

In agent mode — for authenticated agents viewing a consumer's conversation from the agent console — the AI shows an admin overlay: the full lead record, scoring data, and a "Process Purchase" button that charges the consumer's saved payment method. The agent sees the same chat history the consumer does, plus the data layer the consumer doesn't.

This three-mode design shares one AI system prompt and one chat thread, with presentation toggled by a context parameter. It would have been tempting to build three separate chat interfaces — don't. The shared thread means an agent can seamlessly pick up a conversation a consumer started, see the same context, and continue it without asking the consumer to repeat themselves.

## Confirmation Email via Resend

After payment confirmation, the server fires a confirmation email through Resend. The email contains: carrier name, policy number (generated as a UUID and stored in the purchase record), premium amount, coverage summary, and a PDF attachment of the policy terms. The attachment is generated server-side using `pdf-lib` from a template — it's not a real carrier-issued policy document, but it gives the user something tangible to show that the transaction completed.

The email goes to both the purchaser and the agency's general inbox as a BCC. This isn't a Resend limitation — it's a business requirement. Every purchase generates an internal notification. The BCC approach is simpler than a separate notification webhook and keeps the send logic in one place.

## The Design Principle

The insight this project reinforced: purchase friction lives in navigation, not forms. Users don't abandon purchases because the payment form is complex — Stripe's `PaymentElement` is as polished as it gets. They abandon because they were redirected away from the context that motivated the purchase, lost confidence in the middle of the redirect, and didn't come back.

Keeping the user on one surface through the entire flow — recommendation visible, payment form inline, confirmation in-thread — eliminates the friction that comes from navigation, not from payment complexity. The form is three fields. The decision to buy is what's hard. Design around that.
