## The NYS Open Legislation API

NYSgpt was the first project I built in this suite — the origin of the continuous ingest architecture that everything else in the platform would follow. NSRgpt's embedding pipeline, SolarGPT's EIA-923 ingest, the ISO data cron — all of them follow patterns I worked out here first.

The New York State Legislature runs an Open Legislation API that exposes bill data, vote records, committee assignments, and sponsor information. It's genuinely useful — near-real-time updates during session, structured JSON responses, no auth required for basic access. For a policy AI application, it's the closest thing to a primary source you can query programmatically.

The problems are in the details. The data shape is inconsistent across bill types. Senate bills and Assembly bills have slightly different sponsor structures. Memoranda of support and veto messages exist as separate endpoints that aren't always linked to the bill record. Vote records have a different update cadence than bill text. Sponsor information can change if a bill is amended and re-referred. If you ingest once and stop, you quickly drift from the actual legislative record.

NYSgpt runs a sync process that handles all of this — continuously, automatically, and with enough fault tolerance that a brief API outage doesn't corrupt the dataset.

## The Hourly Cron Architecture

The sync runs as a Supabase Edge Function on an hourly cron schedule. The function queries the NYS API for bills updated since the last successful sync timestamp, which we store in a `sync_log` table. This delta approach keeps each sync small — during an active session week, the API might return 50-200 updated bills per hour. During recess, the delta is often zero.

For each updated bill, the sync performs a full replacement of the record rather than a partial update. Legislative records have too many interdependent fields — status, committee, sponsor, floor schedule — to safely merge partial updates without risking inconsistency. A full replacement, inside a transaction, is slower but correct. We use Supabase's `upsert` with `onConflict: 'bill_id'` to handle both new bills and updates.

The function also queues an embedding job for any bill whose text changed since the last sync. Text changes are detected by comparing a hash of the current bill text against the stored hash. If they match, no re-embedding is needed. If they don't, the bill goes into a `pending_embeddings` queue.

## The Embedding Pipeline

Bills are embedded using `text-embedding-3-small` at 256 dimensions, stored in a pgvector column on the bills table. The embedding input is a concatenation of bill number, title, sponsor names, subject matter keywords, and the first 2,000 tokens of bill text. We don't embed the full text because legislative bills can be extremely long — omnibus legislation runs to tens of thousands of words — and embedding the full document produces vectors that are too diffuse to retrieve precisely.

The chunking strategy for longer bills uses the legislative section structure: we embed the preamble, findings, and operative sections separately and store them as child records linked to the parent bill. Retrieval then operates at the section level, which dramatically improves precision for questions about specific provisions within a complex bill.

The `pending_embeddings` queue is processed by a separate Edge Function that runs every 5 minutes. This decouples the sync latency from the embedding latency. If the OpenAI embedding API is slow, bills still sync — they just appear without updated embeddings until the embedding job catches up.

## Sponsor Linkage and Vote Records

Sponsor data deserves special handling. The primary sponsor, multi-sponsors, and co-sponsors are three separate data structures in the NYS API, and the distinction matters for analysis. A bill with 60 co-sponsors is politically different from one with 3. We normalize all three into a `bill_sponsors` junction table with a `sponsor_type` enum.

Vote records are the most fragile part of the pipeline. The API returns vote totals (ayes, nays, excused, abstained) along with individual member votes when available. Individual member votes aren't always available for all chambers and bill types. The sync handles this by storing whatever is available and flagging incomplete vote records with a `vote_completeness` field — complete, totals_only, or unavailable. Downstream queries on vote behavior filter accordingly rather than treating missing data as a vote against.

## The Resync and Re-embedding Scripts

Two shell scripts live in the PolicyGPT repo for manual operations: `resync.sh` and `reembed.sh`. The resync script does a full historical reload — useful when we've changed the schema or discovered a normalization bug that affected a batch of records. It runs the same sync logic but with `since=epoch`, which fetches everything the API has rather than just the delta.

The reembed script processes all bills in the `pending_embeddings` queue regardless of when they were added — useful after a failed embedding run or after we change embedding model versions. Both scripts are idempotent: running them twice produces the same result as running them once.

## Staying Current During Session

The harder problem isn't the sync architecture — it's knowing when the sync is failing silently. A bug in the delta calculation could cause us to stop seeing updates without any visible error. We handle this with a freshness check: a monitoring query runs every 6 hours and verifies that at least one bill was updated in the last 48 hours during active session periods. If not, it triggers an alert and falls back to a full resync for the last 7 days.

This is belt-and-suspenders, but the cost of stale data in a policy application is high enough to justify the redundancy. A user asking about a bill's status and getting an answer based on last week's data doesn't know they're getting stale data — and in policy, timing is often the most important fact.

The delta-sync pattern, the `sync_log` freshness check, and the decoupled embedding queue all became templates I reused in subsequent projects. NSRgpt added pgvector-backed semantic search on top of what I'd built here. SolarGPT applied the same cron-plus-queue structure to EIA-923 plant generation data. The ingest architecture started here, in NYSgpt — the first project — and propagated outward as the suite grew.
