## Why InsuranceGPT, and Not the Others

NYSgpt and NSRgpt are desktop-first research tools. SolarGPT, ChildcareGPT, and SportsGPT came later and didn't need native distribution. InsuranceGPT was the project where mobile reach genuinely mattered — an insurance quote flow needs to reach people on their phones, and App Store presence signals legitimacy to carriers and prospects in a way a mobile website doesn't.

Capacitor is a library from the Ionic team that wraps a web application in a native WebView. The entire mobile packaging layer adds three things to the project: the `@capacitor/core` and `@capacitor/cli` packages, a `capacitor.config.ts` pointing to the Vite build output, and a `build:mobile` script that runs `vite build` with mobile-specific environment variables and then calls `cap sync` to copy the output into the native project directories. The React code doesn't know it's running in a WebView. The Vite build doesn't produce a different bundle. The Capacitor layer is pure packaging.

## The VITE_API_BASE_URL Switch

The one meaningful difference between the web build and the mobile build is the API base URL. When the SPA runs at `insurance.nysgpt.com`, API calls go to `/api/...` — same-origin, no CORS needed. When the same SPA runs inside a Capacitor WebView on a phone, there's no same origin. The WebView loads bundled HTML from the device filesystem, so `/api/quote` resolves to nothing.

The fix is a single environment variable: `VITE_API_BASE_URL`. Empty in the web build. Set to `https://insurance.nysgpt.com` in the mobile build. Every `fetch` call goes through an `apiFetch` wrapper:

```js
const apiFetch = (path, options) =>
  fetch(`${import.meta.env.VITE_API_BASE_URL || ''}${path}`, options);
```

The `build:mobile` script sets the variable inline:

```json
"build:mobile": "VITE_API_BASE_URL=https://insurance.nysgpt.com vite build"
```

One variable, one wrapper function, no mobile-specific code anywhere else.

## cap sync and the Native Directories

After `build:mobile` completes, `cap sync` copies the Vite output into `ios/App/App/public/` and `android/app/src/main/assets/public/`. Xcode and Android Studio treat the project as a standard native app from there.

I don't commit the `ios/` and `android/` directories. They're generated artifacts — fully regenerable by running `npx cap add ios`, `npx cap add android`, then `build:mobile && cap sync`. Committing them adds thousands of files that change on every Capacitor version update and produce diffs with no meaningful content.

## What Breaks at Scale

**Authentication:** The mobile WebView shares no cookie jar with Safari. InsuranceGPT uses JWTs in localStorage, which survives the WebView context cleanly. If you rely on httpOnly cookies, mobile is a harder problem.

**App Store Review:** Apple scrutinizes WebView-based apps. An app that's a pure website wrapper won't pass. InsuranceGPT passed because the quote flow and agent console provide genuine mobile utility.

**Updates:** Web versions update on deploy. Mobile versions update through the App Store — users on old versions hit the current API. API versioning and backward compatibility become concerns earlier than they would for web-only.

## When This Approach Makes Sense

Capacitor's packaging is the right call when you have a Vite/React SPA already working on the web, a small team that can't staff separate iOS and Android engineers, and functionality that maps well to mobile without deep native integration. It's the wrong call when key interactions depend on native gestures, offline-first data sync, or rendering fidelity that WebView can't match.

For InsuranceGPT — a form-heavy, conversation-driven app where the value is the AI recommendation layer — the WebView performs indistinguishably from native. The meaningful experience is the data and the AI, not the rendering layer. No subsequent project in the portfolio needed this, which is why it stays here.
