## The Setup

Capacitor is a library from the Ionic team that wraps a web application in a native WebView and gives it access to native device APIs. For InsuranceGPT, the use case is simpler than that: I don't need camera access or push notifications. I need the app to be installable on iOS and Android and to look like a native app in the app store listing.

The entire mobile packaging layer adds three things to the project: the `@capacitor/core` and `@capacitor/cli` packages, a `capacitor.config.ts` file that points to the Vite build output directory, and a `build:mobile` script that runs `vite build` with mobile-specific environment variables and then calls `cap sync` to copy the output into the native project directories.

That's the whole integration. The React code doesn't know it's running in a WebView. The Vite build doesn't produce a different bundle. The Capacitor layer is pure packaging.

## The VITE_API_BASE_URL Switch

The one meaningful difference between the web build and the mobile build is the API base URL. When the SPA runs at `insurance.nysgpt.com`, API calls go to `/api/...` — same-origin requests, no CORS configuration needed. When the same SPA runs inside a Capacitor WebView on a phone, there's no "same origin." The WebView loads the bundled HTML from the device filesystem, so `/api/quote` resolves to nothing.

The fix is a single environment variable: `VITE_API_BASE_URL`. In the web build, it's empty — the app uses relative paths. In the mobile build, it's set to `https://insurance.nysgpt.com` — the app constructs absolute URLs. Every `fetch` call in the codebase goes through a small `apiFetch` wrapper that prepends `VITE_API_BASE_URL` to the path:

```js
const apiFetch = (path, options) =>
  fetch(`${import.meta.env.VITE_API_BASE_URL || ''}${path}`, options);
```

The `build:mobile` script in `package.json` sets the variable inline:

```json
"build:mobile": "VITE_API_BASE_URL=https://insurance.nysgpt.com vite build"
```

One variable, one wrapper function, no mobile-specific code anywhere else in the codebase.

## cap sync and the Native Directories

After `build:mobile` completes, `cap sync` copies the Vite output into `ios/App/App/public/` and `android/app/src/main/assets/public/`. From there, Xcode and Android Studio treat the project as a standard native app — they just happen to be rendering a WebView pointed at a local HTML file.

`cap sync` also handles plugin synchronization: if you add a Capacitor plugin (camera, geolocation, etc.), `cap sync` installs the native implementation into both platform directories automatically. Since InsuranceGPT doesn't use any native APIs, `cap sync` is purely a file copy step, but the mechanism is the same.

I do not commit the `ios/` and `android/` directories to the repository. They're in `.gitignore`. The reasoning: these directories are generated artifacts that can be fully regenerated by running `npx cap add ios` and `npx cap add android` followed by `build:mobile && cap sync`. Committing them adds thousands of files — Xcode project metadata, Gradle wrapper files, compiled dependency artifacts — that change on every Capacitor version update and create enormous diffs with no meaningful content. Any developer who needs to build the mobile app regenerates them locally.

## What Breaks at Scale

This approach works well for a small team building a single SPA with a remote API backend. It starts to show cracks in a few scenarios:

**Authentication:** The mobile WebView shares no cookie jar with Safari. Session cookies set by the web app don't persist correctly in some WebView configurations. InsuranceGPT uses JWTs stored in localStorage, which survives the WebView context cleanly. If you rely on httpOnly cookies for auth, mobile is a more complicated problem.

**App Store Review:** Apple's App Store review team scrutinizes WebView-based apps closely. If the app doesn't provide meaningful native functionality beyond what the website offers, it gets rejected under the "web content" guidelines. InsuranceGPT passed review because the quote flow and agent console are genuinely useful in a native context — notifications and offline support are on the roadmap. An app that's purely a WebView wrapper for a marketing site won't pass.

**Updates:** The web version updates instantly on deploy. The mobile version requires the user to update through the App Store, which means there can be active users on old versions of the app hitting the current API. You need to think about API versioning and backward compatibility earlier than you would for web-only.

## When This Approach Makes Sense

Capacitor's WebView packaging is the right call when: you have a Vite/React SPA that's already working well on the web, you have a small team that can't staff separate iOS and Android engineers, and the app's functionality maps well to a mobile use case without requiring deep native integration.

It's the wrong call when: your app's key interactions depend on native gestures or sensors, you need offline-first data sync, or you're targeting users who are performance-sensitive enough to notice WebView rendering fidelity differences compared to native UIKit or Jetpack Compose.

For InsuranceGPT — a form-heavy, conversation-driven application where the core value is the AI recommendation layer — the WebView performs indistinguishably from native. Users getting quotes and comparing carriers don't care whether the input is a UITextField or an HTML input. The meaningful experience is the data and the AI, not the rendering layer.
