## The Constraint

Vercel's hobby plan limits you to 12 serverless functions. This isn't a soft limit — deploy the 13th and the build fails. InsuranceGPT was my third project, after NYSgpt (bills/members/committees) and NSRgpt. Those first two were read-heavy apps with thin backends. InsuranceGPT was different: it added commerce (Stripe), drip campaigns, multi-role auth, and AI outreach in a single deployment. The initial design called for 15 separate API routes and immediately hit the wall.

Before hitting the limit I hadn't thought about function count. After hitting it I had to audit every endpoint and ask: does this actually need to be its own function, or am I cargo-culting REST conventions that don't serve a purpose here?

## The Consolidations

The first easy win was NHTSA lookup and SaferCar ratings. Both are proxy calls against NHTSA APIs. The frontend always calls both in sequence when a VIN is entered — there's no case where you'd want one without the other. I merged them into a single `/api/vehicle` endpoint with a `type` query parameter: `?type=vin` for vehicle identity, `?type=ratings` for safety data. One function, two sources, no behavior change for the frontend.

The second consolidation was more significant: lead ingest, lead scoring, drip enqueue, and LMS forwarding collapsed into a single `/api/leads` POST handler. The original design chained these as four separate functions. That meant four cold starts, four network hops, and four independent failure surfaces — the downstream caller never heard from if something failed mid-chain. The consolidated handler runs all four operations in a single invocation and returns one acknowledgment. Failures surface in one place.

## The Full Function Inventory

After consolidation, the 12-function inventory is:

1. `/api/vehicle` — NHTSA VIN lookup + SaferCar ratings (type param)
2. `/api/quote` — AI-generated carrier comparison + pricing
3. `/api/leads` — ingest + scoring + drip enqueue + LMS forward
4. `/api/checkout` — Stripe PaymentIntent creation
5. `/api/confirm` — payment confirmation + policy record write
6. `/api/auth/consumer` — consumer login + session issue
7. `/api/auth/agent` — agent login + session issue
8. `/api/auth/admin` — admin login + elevated session
9. `/api/auth/reset` — password reset token issue + email send
10. `/api/carriers` — carrier catalog + pricing tiers fetch
11. `/api/admin/dashboard` — aggregate stats for admin view
12. `/api/health` — build verification + DB ping

The health check almost got cut, but a single DB ping endpoint that returns 200/503 based on Neon connection status earned its slot for deployment verification.

## Why the Constraint Was a Feature

The consolidation forced me to think clearly about what a function boundary is actually for. In a monolith you'd have a `LeadService` class with four methods. In serverless, a function is closer to a deployed service — its own cold start, its own failure domain, its own billing unit. Splitting lead ingest from lead scoring was always the wrong call; they're steps in a single transaction, not separate services.

Without the constraint I'd have shipped something that looked cleanly separated but was just latency and complexity. Every project that came after InsuranceGPT started with tighter API boundaries because of this lesson. The pattern I now apply: before adding a serverless function, ask whether the new behavior is a genuinely separate concern or just a new method on an existing one. Twelve functions is a tight budget — it's also a forcing function for a cleaner API surface.
