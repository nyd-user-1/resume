## The Fork-First Approach

I didn't set out to build a design system. I set out to build NYSgpt — a legislative research app covering New York State bills, members, and committees. That first app became the mothership: hundreds of components, experiments, layout patterns, and index+detail page flows accumulated over the first two months. When NSRgpt came next, the fastest path was forking NYSgpt, stripping what didn't apply, and building on what did.

That fork contained layout scaffolding, route structure, CSS variables, a sidebar component, a loading skeleton pattern, and about a dozen utility functions accumulated while debugging NYSgpt. I didn't curate any of it deliberately — the fork brought it wholesale. Within a week, about 40% was deleted as NYSgpt-specific, another 30% worked without modification, and the remaining 30% had to evolve to fit NSRgpt's different domain.

That 30% is where the actual design language came from. Not from planning, not from a design sprint, but from components that got used in a new context and had to generalize to survive.

The sequence: NYSgpt → NSRgpt → InsuranceGPT → SolarGPT → ChildcareGPT → SportsGPT. Each fork carried forward what survived from the previous one and contributed new patterns back into the shared vocabulary. SportsGPT — the most refined app — came last. It didn't originate the design language; it inherited and sharpened it.

## AppPanel: From Insurance Detail Sidebar to Universal Detail Shell

The first component to survive the full chain of copying was AppPanel. The concept originated in InsuranceGPT, where a detail page had a button group of three buttons — outreach scripts, SMS, and AI chat — each triggering a panel that slid in from the right side. Clicking didn't navigate away. The underlying detail page stayed fully visible and interactive. That was the first time the side panel was a deliberate interaction pattern rather than an afterthought.

When SolarGPT needed a panel to display county electricity profiles when a user clicked a map region, the InsuranceGPT panel pattern translated directly. But county profiles contain more content than an insurance outreach script, so a `size` prop was added: `"sm"` for the original third-width, `"md"` for half-width. That was the first time AppPanel became genuinely abstract rather than purpose-specific.

SportsGPT pushed it further: parallel routes made the panel server-rendered and URL-shareable. AppPanel split into a client shell and a server-rendered interior via a Next.js `@panel` slot — a better architecture than would have been designed upfront, because it was solving a real problem rather than anticipating one.

## InfoTable: From InsuranceGPT Detail Pages to Every Profile in the Suite

InfoTable's design crystallized in InsuranceGPT. The detail page pattern established there — an ~866px container, H1 with a button group opposite, breadcrumbs, a hero image, then H2 headings each followed by an InfoTable — became the template every subsequent app copied. The component showed labeled fact rows organized into grouped sections: policy details, coverage terms, provider information.

NYSgpt had a simpler ancestor for bill metadata display, but InsuranceGPT is where the grouped-section, null-safe, responsive accordion form of InfoTable solidified into something worth carrying forward.

SolarGPT adopted InfoTable for county and state electricity profiles (generation capacity, emissions rates, grid participation by fuel type). ChildcareGPT used it for facility profiles — licensing status, capacity, inspection history, contact information. The data shapes were completely different, but the component handled them without modification after the domain-specific types were replaced with a generic `{ label: string; value: string | number | null; group?: string }` row structure. PolicyGPT used it for bill metadata. NSRgpt used it for filing metadata in document panels.

The component has now appeared in five of the six apps without structural changes — only the data shapes passed into it have changed. The generalization happened organically across three copying events, each one revealing which assumptions were domain-specific and which were structural.

## RevealFx: The Animation That Spread Everywhere

RevealFx is a mask-and-blur entrance animation — elements clip-path from 0% to 100% with a simultaneous blur fade as they enter the viewport. It started in SportsGPT's team hero sections, where the school photo and team name should feel like they were materializing rather than just appearing.

I used an Intersection Observer with a CSS custom property to drive the animation — no external library, just a small hook and a few lines of CSS. When I copied it into SolarGPT's county detail pages, I realized the hook was completely portable. When I added it to ChildcareGPT's facility pages, I extracted it into a standalone `RevealFx` component with a `delay` prop for staggering multiple elements. Every subsequent detail page in every app now uses it.

What makes RevealFx a good example of how shared components develop is that it never changed structurally after the ChildcareGPT extraction. The API settled on the first real generalization: `children`, `delay`, `className`. Nothing was added for subsequent apps. When a component reaches that kind of stability quickly, it means the abstraction found the right boundary.

## Why Suite-First Beats System-First

The conventional wisdom is that you should define your design system before building products. Extract components, document them, publish them as a package, then build on top. That approach has real advantages for large teams working on many products simultaneously.

For a solo developer building sequentially, it inverts the value. You don't know which components will survive until you've tried to copy them. A design system built before you've used its components in anger is full of abstractions that feel right but aren't — too many props that never get used, too little flexibility where flexibility actually matters, too many opinions baked in before you've seen the edge cases.

The fork-first approach lets the suite produce the design system rather than the other way around. Components earn their place in the shared library by surviving a copying event and proving they work in a second context. The ones that don't survive — that need to be rewritten from scratch for each new app — tell you something equally important: that piece of UI is domain-specific and shouldn't be abstracted.

After six apps, the shared component library has about twenty components in it. Every one of them passed the copying test at least twice. None of them were designed to be reusable before they were useful. NYSgpt laid the foundation; each subsequent fork both consumed and contributed to it.
